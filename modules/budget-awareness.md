
# 成本感知开发

## 概述

通过模型路由、预算跟踪、重试逻辑和提示缓存来控制 LLM API 成本，同时保持质量。

**核心原则：** 从最便宜的模型开始。仅在复杂度需要时才升级。

## 何时使用

- 构建调用 Claude、OpenAI 或类似 LLM API 的应用程序
- 处理成本快速累积的批量项目
- 在固定的 API 预算内工作
- 在不牺牲质量的情况下优化复杂任务的成本

## 四种技术

### 1. 按复杂度路由模型

自动为简单任务选择更便宜的模型。为复杂任务保留昂贵的模型。

**复杂度信号：**
- 短输入（<10K 字符），少量项目（<30）→ **Haiku**（最便宜）
- 中等复杂度 → **Sonnet**（标准）
- 长输入、多项目、设计判断 → **Opus**（最强大）

**价格参考（2025-2026）：**

| 模型 | 输入（美元/百万令牌） | 输出（美元/百万令牌） | 相对成本 |
|-------|---------------------|----------------------|----------|
| Haiku 4.5 | $0.80 | $4.00 | 1x |
| Sonnet 4.6 | $3.00 | $15.00 | ~4x |
| Opus 4.5 | $15.00 | $75.00 | ~19x |

### 2. 不可变成本追踪

使用冻结的数据类跟踪累计支出。每个 API 调用返回新的追踪器 — 永远不改变状态。

### 3. 窄范围重试逻辑

仅在暂时性错误时重试。认证或错误请求错误快速失败。

**仅重试：** `APIConnectionError`、`RateLimitError`、`InternalServerError`
**立即失败：** `AuthenticationError`、`BadRequestError`

### 4. 提示缓存

缓存长的系统提示词，避免在每个请求上重新发送。对超过 1024 个令牌的系统提示词使用 `cache_control: {"type": "ephemeral"}`。

## 组合流水线

```
1. 路由模型 → select_model(text_length, item_count)
2. 检查预算 → if tracker.over_budget: raise BudgetExceeded
3. 带重试调用 → call_with_retry(lambda: client.messages.create(...))
4. 追踪成本 → tracker = tracker.add(record)
```

## 最佳实践

- **从便宜开始**，仅在达到复杂度阈值时升级
- **在处理批次前设置明确的预算限制**
- **记录模型选择决策**，以便根据实际数据调整阈值
- **对超过 1024 个令牌的系统提示词使用提示缓存**
- **永远不在认证或验证错误时重试**

## 反模式

- 无论复杂度如何都使用最昂贵的模型
- 重试所有错误（在永久性故障上浪费预算）
- 改变成本追踪状态（使调试和审计变得困难）
- 在整个代码库中硬编码模型名称
- 对重复的系统提示词忽略提示缓存

## 集成

**相关技能：**
- **ultrapowers:delegate-and-verify** — 派发子代理时使用模型路由（机械任务用便宜模型，设计/架构用强大模型）
- **ultrapowers:parallel-dispatch** — 追踪每个代理的成本以避免预算超支
