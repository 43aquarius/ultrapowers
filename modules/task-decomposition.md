
# 编写计划

## 概述

编写全面的实施计划，假设工程师对我们的代码库零上下文且品味存疑。记录他们需要知道的一切：每个任务要接触哪些文件、代码、测试、他们可能需要检查的文档、如何测试。以小型任务的形式给他们整个计划。DRY。YAGNI。TDD。频繁提交。

假设他们是有技能的开发人员，但几乎不了解我们的工具集或问题域。假设他们不太了解良好的测试设计。

**开始时声明：** "我正在使用 writing-plans 技能来创建实施计划。"

**上下文：** 如果在隔离的工作树中工作，它应该是在执行时通过 `ultrapowers:branch-isolation` 技能创建的。

**计划保存到：** `docs/ultrapowers/plans/YYYY-MM-DD-<feature-name>.md`
- （用户对计划位置的偏好覆盖此默认值）

## 范围检查

如果规范涵盖多个独立子系统，它应该在头脑风暴期间被分解为子项目规范。如果没有，建议将其分解为单独的计划 — 每个子系统一个。每个计划应该产生可独立工作的、可测试的软件。

## 文件结构

在定义任务之前，规划将创建或修改哪些文件以及每个文件的职责。这是分解决策被锁定的地方。

- 设计具有清晰边界和良好定义的接口的单元。每个文件应该有一个明确的职责。
- 你对你一次能保持在上下文中的代码推理最好，当文件聚焦时你的编辑更可靠。偏好较小的、聚焦的文件，而不是做得太多的大文件。
- 一起更改的文件应该放在一起。按职责拆分，而不是按技术层。
- 在现有代码库中，遵循已建立的模式。如果代码库使用大文件，不要单方面重构 — 但如果你正在修改的文件已经变得难以处理，在计划中包含拆分是合理的。

此结构通知任务分解。每个任务应该产生独立的有意义的更改。

## 小型任务粒度

**每个步骤是一个操作（2-5 分钟）：**
- "写失败的测试" — 步骤
- "运行它确保失败" — 步骤
- "实施使测试通过的最简代码" — 步骤
- "运行测试确保通过" — 步骤
- "提交" — 步骤

## 计划文档标题

**每个计划必须以此标题开始：**

```markdown
# [功能名称] 实施计划

> **对于代理工作者：** 必需子技能：使用 ultrapowers:delegate-and-verify（推荐）或 ultrapowers:plan-execution 来逐个任务实施此计划。步骤使用复选框（`- [ ]`）语法进行跟踪。

**目标：** [一句话描述构建什么]

**架构：** [2-3 句话关于方法]

**技术栈：** [关键技术/库]

---
```

## 任务结构

````markdown
### 任务 N：[组件名称]

**文件：**
- 创建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

- [ ] **步骤 1：写失败的测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试验证它失败**

运行：`pytest tests/path/test.py::test_name -v`
预期：FAIL，"function not defined"

- [ ] **步骤 3：写最简实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试验证它通过**

运行：`pytest tests/path/test.py::test_name -v`
预期：PASS

- [ ] **步骤 5：提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

## 无占位符

每个步骤必须包含工程师需要的实际内容。这些是**计划失败** — 永远不要写它们：
- "TBD"、"TODO"、"稍后实现"、"填写详细信息"
- "添加适当的错误处理" / "添加验证" / "处理边缘情况"
- "为上述内容编写测试"（没有实际测试代码）
- "类似于任务 N"（重复代码 — 工程师可能不按顺序阅读任务）
- 描述做什么而不显示如何做的步骤（代码步骤需要代码块）
- 在任何任务中未定义的类型、函数或方法的引用

## 记住
- 始终精确的文件路径
- 每个步骤中的完整代码 — 如果步骤更改代码，显示代码
- 精确的命令和预期输出
- DRY、YAGNI、TDD、频繁提交

## 自我审查

写完完整计划后，用新鲜的眼光查看规范并对照它检查计划。这是你自己运行的清单 — 不是子代理派发。

**1. 规范覆盖：** 浏览规范中的每个部分/需求。你能指出实施它的任务吗？列出任何差距。

**2. 占位符扫描：** 搜索计划中的危险信号 — 上述"无占位符"部分中的任何模式。修复它们。

**3. 类型一致性：** 你在后面任务中使用的类型、方法签名和属性名称是否与你在前面任务中定义的匹配？在任务 3 中称为 `clearLayers()` 但在任务 7 中称为 `clearFullLayers()` 的函数是一个 bug。

如果你发现问题，内联修复它们。无需重新审查 — 只需修复并继续。如果你发现没有任务对应的规范需求，添加任务。

## 执行交接

保存计划后，提供执行选择：

**"计划完成并保存到 `docs/ultrapowers/plans/<filename>.md`。两个执行选项：**

**1. 子代理驱动（推荐）** — 我每个任务派发一个新的子代理，任务之间审查，快速迭代

**2. 内联执行** — 使用 executing-plans 在此会话中执行任务，带检查点的批量执行

**选择哪个？"**

**如果选择子代理驱动：**
- **必需子技能：** 使用 ultrapowers:delegate-and-verify
- 每个任务新子代理 + 两阶段审查

**如果选择内联执行：**
- **必需子技能：** 使用 ultrapowers:plan-execution
- 带审查检查点的批量执行
