
# 评估驱动开发

## 概述

在编写代码**之前**，将预期行为定义为可执行评估。运行基线（应该失败）。实现。重新运行以验证通过。追踪 pass@k 指标来衡量可靠性。

**核心原则：** 如果你无法衡量成功，你就不能声称成功。

**为什么对 AI 生成代码特别重要：** 当同一个模型编写和审查自己的代码时，存在系统性盲点。评估能捕捉到自我审查遗漏的问题。

## 何时使用

**在以下情况使用：**
- 正确性可以衡量（测试、lint、类型检查、API 响应）
- AI 生成实现（盲点风险更高）
- 回归预防很重要
- 需要比较方案或模型

**在以下情况不使用：**
- 输出纯粹是创意/主观的（设计、写作）
- 不存在可衡量的成功标准
- 一次性脚本，用完即弃

## 流程

```
1. 定义  → 编写编码预期行为的评估
2. 基线  → 运行评估（预期失败）
3. 实现  → 编写代码使评估通过
4. 验证  → 运行评估（预期通过）
5. 回归  → 还原修复 → 运行（必须失败）→ 恢复 → 运行（通过）
```

### 第 1 步：定义评估

编写编码预期行为的评估：

```yaml
# evals/feature-name.yaml
name: "功能名称评估"
evals:
  - name: "核心行为有效"
    type: pytest
    command: "pytest tests/test_feature.py::test_core -x"
    expected: "pass"

  - name: "边界情况：空输入"
    type: pytest
    command: "pytest tests/test_feature.py::test_empty -x"
    expected: "pass"

  - name: "无回归"
    type: pytest
    command: "pytest tests/ -x --tb=no -q"
    expected: "0 failures"
```

**评估器类型：**
- **基于 pytest：** 确定性，优先选择。运行实际测试。
- **grep 模式：** 检查输出中的预期模式。
- **LLM 作为裁判：** 用于主观质量（谨慎使用，成本高）。

### 第 2 步：基线

在编写任何代码之前运行所有评估：

```bash
# 运行基线
pytest tests/test_feature.py -x
# 预期：失败（功能尚不存在）

# 记录基线指标
echo "基线：0/3 个评估通过"
```

**这至关重要。** 没有基线，你无法知道你的实现是否真正改变了什么。

### 第 3 步：实现

编写代码使评估通过。在此框架内遵循 TDD：

- 每个评估应有对应的失败测试
- 编写测试 → 看它失败 → 编写最小代码 → 看它通过
- 使用 `ultrapowers:red-green-cycle` 进行 TDD 循环

### 第 4 步：验证

实现后运行所有评估：

```bash
pytest tests/test_feature.py -x
# 预期：通过

# 记录实现后指标
echo "实现后：3/3 个评估通过"
```

### 第 5 步：回归验证

证明评估确实在测试什么：

```
对于每个评估：
  1. 还原代码更改
  2. 运行评估 → 必须失败
  3. 恢复代码更改
  4. 运行评估 → 必须通过
```

如果一个评估在更改前后都通过，那它测试的不是你以为的内容。

## Pass@K 指标

对于关键功能，运行多次试验以捕捉方差：

| 指标 | 含义 | 目标 |
|------|------|------|
| pass@1 | 首次尝试成功率 | ≥ 80% |
| pass@3 | 3 次内成功率 | ≥ 95% |
| pass^3 | 3 次全部成功 | ≥ 70% |

**何时使用 pass@k：**
- 比较模型或方案
- 可靠性至关重要的关键功能
- 衡量技能有效性

## AI 回归模式

AI 编写和审查自己代码时的常见盲点：

| 模式 | 症状 | 预防 |
|------|------|------|
| 沙箱/生产路径不匹配 | 隔离测试通过，集成失败 | 使用真实依赖测试 |
| SELECT 子句遗漏 | 返回所有列而非指定列 | 测试中显式断言列 |
| 错误状态泄漏 | 正常路径有效，错误路径静默失败 | 先写错误情况测试 |
| 乐观更新缺少回滚 | UI 显示成功，后端失败 | 测试回滚路径 |
| 日期格式假设 | 假设 ISO-8601，数据使用不同格式 | 测试中显示实际数据格式 |

## 集成

**必需的工作流技能：**
- **ultrapowers:red-green-cycle** — 用于每个评估内的红绿循环
- **ultrapowers:pre-completion-check** — 在声称成功之前验证评估结果
- **ultrapowers:root-cause-analysis** — 当评估失败需要根因分析时

**相关 ECC 技能：**
- `eval-harness` — 带有评分器类型的正式评估框架
- `agent-eval` — 带有通过率、成本、时间指标的代理对比
- `ai-regression-testing` — 基于 bug 的测试编写方法

## 危险信号

- 在实现后编写评估（违背目的）
- 始终通过的评估（没有测试任何东西）
- 跳过基线运行
- 跳过回归验证（还原 → 失败 → 恢复 → 通过）
- 仅使用 LLM 作为裁判（昂贵、不一致）
- 不追踪关键功能的 pass@k

## 底线

**评估是带有基线和验证步骤的测试。**

定义 → 基线（失败）→ 实现 → 验证（通过）→ 回归检查。

如果跳过基线，你在猜测。如果跳过回归验证，你不知道你的测试是否有意义。
