
# 完成前验证

## 概述

没有验证就声称工作完成是不诚实，不是效率。

**核心原则：** 先有证据再有断言，始终。

**违反此规则的字面意思就是违反此规则的精神。**

## 铁律

```
没有新鲜验证证据，不允许声称完成
```

如果你在这条消息中没有运行验证命令，你不能声称它通过。

## 门控函数

```
在声称任何状态或表达满意之前：

1. 识别：什么命令证明这个声明？
2. 运行：执行完整命令（新鲜的，完整的）
3. 阅读：完整输出，检查退出代码，计算失败
4. 验证：输出确认声明了吗？
   - 如果否：用证据陈述实际状态
   - 如果是：用证据陈述声明
5. 只有那时：做出声明

跳过任何步骤 = 说谎，不是验证
```

## 常见失败

| 声明 | 需要 | 不充分 |
|-------|----------|----------------|
| 测试通过 | 测试命令输出：0 个失败 | 之前的运行，"应该通过" |
| 检查器干净 | 检查器输出：0 个错误 | 部分检查，外推 |
| 构建成功 | 构建命令：退出 0 | 检查器通过，日志看起来不错 |
| Bug 修复 | 测试原始症状：通过 | 代码更改，假设已修复 |
| 回归测试有效 | 红绿循环验证 | 测试通过一次 |
| 代理完成 | VCS diff 显示更改 | 代理报告"成功" |
| 需求满足 | 逐行检查清单 | 测试通过 |

## 危险信号 — 停止

- 使用"应该"、"可能"、"似乎是"
- 在验证之前表达满意（"太好了！"、"完美！"、"完成！"等）
- 在没有验证的情况下即将提交/push/PR
- 信任代理成功报告
- 依赖部分验证
- 想"就这一次"
- 累了想结束工作
- **任何暗示成功的措辞，而没有运行验证**

## 借口预防

| 借口 | 现实 |
|--------|---------|
| "现在应该有效" | 运行验证 |
| "我有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "检查器通过了" | 检查器 ≠ 编译器 |
| "代理说成功了" | 独立验证 |
| "部分检查就够了" | 部分证明不了什么 |
| "不同的词所以规则不适用" | 精神高于字面 |

## 编辑前的事实门控（来自 gateguard）

LLM 自我评估不起作用。问"你确定吗？"总是得到"确定"。但要求*具体事实*会迫使调查，而调查本身创造了改变输出质量的上下文。

**编辑的三阶段门控：**

```
1. 拒绝  — 阻止首次编辑/写入/Bash 尝试
2. 强制  — 明确告知模型需要收集哪些事实
3. 允许  — 在事实呈现后允许重试
```

**编辑文件前，要求这些事实：**

```
在编辑 {file_path} 之前，请先呈现以下事实：

1. 列出所有导入/引用此文件的文件（使用 Grep）
2. 列出受此更改影响的公共函数/类
3. 如果此文件读取/写入数据文件，请显示字段名称、结构以及日期格式（使用脱敏值，而非原始生产数据）
4. 逐字引用用户当前的指令
```

**创建新文件前：**

```
在创建 {file_path} 之前，请先说明以下事实：

1. 命名将调用此新文件的文件及行号
2. 确认没有现有文件具有相同功能（使用 Glob）
3. 如果此文件读取/写入数据文件，请展示字段名称、结构及日期格式
4. 逐字引用用户当前的指令
```

**破坏性 Bash 前（rm -rf、git push --force、DROP TABLE 等）：**

```
1. 列出此命令将修改或删除的所有文件/数据
2. 编写一行回滚步骤
3. 逐字引用用户当前的指令
```

**证据：** 相同代理、相同任务的 A/B 测试 — 有门控 9.0 vs 无门控 6.75 平均质量分数（+2.25）。两个代理生成的代码都能运行并通过测试。区别在于设计深度。

**反模式：** 不要用自我评估替代。"你确定吗？"总是得到"确定。"调查创造上下文；自我评估不创造。

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] "所有测试通过"
❌ "现在应该通过" / "看起来正确"
```

**回归测试（TDD 红绿）：**
```
✅ 编写 → 运行（通过）→ 还原修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我写了回归测试"（没有红绿验证）
```

**构建：**
```
✅ [运行构建] [看到：退出 0] "构建通过"
❌ "检查器通过了"（检查器不检查编译）
```

**需求：**
```
✅ 重读计划 → 创建清单 → 验证每个 → 报告差距或完成
❌ "测试通过，阶段完成"
```

**代理委派：**
```
✅ 代理报告成功 → 检查 VCS diff → 验证更改 → 报告实际状态
❌ 信任代理报告
```

## 为什么这很重要

来自 24 条失败记忆：
- 你的合作伙伴说"我不相信你" — 信任破裂
- 未定义函数发布了 — 会崩溃
- 缺少需求发布了 — 功能不完整
- 在错误完成上浪费时间 → 重定向 → 返工
- 违反："诚实是核心价值。如果你撒谎，你将被替换。"

## 何时应用

**始终在以下之前：**
- 任何成功/完成声明的变体
- 任何满意的表达
- 关于工作状态的任何正面陈述
- 提交、PR 创建、任务完成
- 移动到下一个任务
- 委派给代理

**规则适用于：**
- 精确短语
- 改述和同义词
- 成功的暗示
- 任何暗示完成/正确的交流

## 底线

**验证没有捷径。**

运行命令。阅读输出。**然后**声明结果。

这是不可商量的。
