# Ultrapowers — 作品说明

## 一、作品概述

### 1.1 作品名称

**Ultrapowers — AI 编程全生命周期技能集**

### 1.2 作品定位

Ultrapowers 是一个面向 AI 编程助手（Claude Code、Copilot CLI、Gemini CLI 等）的**单体路由式技能集**。它将软件开发过程中反复验证的工作模式系统化为一组**可路由、可组合、防弹化**的子技能，覆盖从会话启动到代码交付的完整开发生命周期。

### 1.3 核心理念

> **先有证据，再有断言。先写测试，再写代码。先找根因，再提修复。**

这三条原则不是口号——它们被编码为 18 个子技能中的"铁律"和"反模式"，通过命令式语言和结构性防御确保 AI 在压力下也能遵循工程纪律。

### 1.4 规模

| 指标 | 数量 |
|------|------|
| 子技能模块 | 18 个 |
| 子代理提示模板 | 6 个 |
| 参考文档 | 9 个（含代码示例） |
| 辅助脚本 | 4 个 |
| 深度技术文档 | 12 篇 |
| 总代码/文本量 | ~8500 行 |

## 二、问题背景与动机

### 2.1 问题：AI 编程助手缺乏工程纪律

当前 AI 编程助手（如 Claude Code）在实现具体任务时表现出色，但在**工程纪律**方面存在系统性缺失：

1. **跳过设计阶段**：AI 倾向于直接写代码，而非先理解需求和设计方案
2. **测试滞后**：测试通常在实现之后编写，甚至不写
3. **症状修复**：遇到 bug 时直接修补表面症状，而非追溯根因
4. **缺乏验证**：声称"完成"时没有系统化的验证流程
5. **知识不沉淀**：每次会话的经验无法复用到下次

这些问题不是 AI 能力的缺失，而是**缺乏结构化的工作流约束**。

### 2.2 现有解决方案的不足

- **系统提示词（System Prompt）**：过于笼统，无法覆盖具体场景
- **CLAUDE.md 规则**：线性列表，缺乏路由机制和条件触发
- **独立技能（Skills）**：功能单一，无法覆盖完整开发生命周期
- **人工 Code Review**：无法实时介入 AI 的编码过程

### 2.3 我们的解决方案

Ultrapowers 将完整的开发生命周期编码为**分层路由的技能集**，让 AI 在每一步都有明确的指令、铁律和反模式约束。

## 三、技术架构

### 3.1 分层路由模型

Ultrapowers 的核心创新是**分层路由**设计。`SKILL.md` 作为唯一的入口路由器，根据当前任务类型将执行分发到 `modules/` 下的对应子模块：

```
用户请求
    │
    ▼
┌──────────────────────┐
│     SKILL.md         │  ← 入口路由器：判断任务类型，查注册表
│   （技能注册表）       │
└──────┬───────────────┘
       │
       ├── 启动层 ──→ skill-bootstrapping.md
       ├── 创意层 ──→ ideation.md
       ├── 规划层 ──→ task-decomposition.md
       ├── 执行层 ──→ delegate-and-verify.md / plan-execution.md
       ├── 质量层 ──→ red-green-cycle.md / root-cause-analysis.md / ...
       ├── 集成层 ──→ branch-isolation.md / branch-wrap-up.md
       ├── 优化层 ──→ parallel-dispatch.md / budget-awareness.md
       └── 知识层 ──→ knowledge-capture.md / skill-authoring.md / skill-discovery.md
```

**路由优势**：
- **按需加载**：只读取当前任务需要的模块，避免上下文窗口浪费
- **关注点分离**：每个子技能聚焦单一职责，指令精确不模糊
- **可扩展**：新增技能只需添加模块文件并在注册表登记

### 3.2 八大层级 · 18 项技能

#### 启动层（1 项）

**Skill Bootstrapping**：会话开始时建立指令优先级、平台适配、安全防护和上下文预算意识。包含 12 种"危险信号"检测——当 AI 产生"这只是个简单问题"等想法时自动阻止合理化。

#### 创意层（1 项）

**Ideation**：动手写代码前的强制设计阶段。9 步清单确保从项目背景探索到方案批准的全流程。核心铁律：

> 在展示设计并获得用户批准之前，不得调用任何实现技能、编写任何代码。

#### 规划层（1 项）

**Task Decomposition**：将设计文档分解为有序的实施计划，每个任务具备目标、文件清单、依赖关系和验收标准。

#### 执行层（2 项）

**Delegate & Verify**：为每个任务派发独立子代理，每个任务后进行两阶段审查（规范符合性 + 代码质量）。子代理获得精确构建的上下文，不继承会话历史。

**Plan Execution**：在独立会话中从头到尾执行完整计划。

#### 质量层（6 项）

| 技能 | 铁律 |
|------|------|
| Red-Green Cycle（TDD） | 没有失败的测试，不允许写生产代码 |
| Root-Cause Analysis（调试） | 没有根本原因调查，不允许修复 |
| Code Review Request | 派发独立审查代理，聚焦工作成果 |
| Review Feedback | 逐项理解、分类、回应审查意见 |
| Pre-Completion Check | 声称完成前的最终验证清单 |
| Metric-Driven Dev | 用可量化指标验证正确性 |

#### 集成层（2 项）

**Branch Isolation**：使用 Git Worktree 创建隔离工作区。
**Branch Wrap-Up**：验证完整性、清理、合并、更新文档。

#### 优化层（2 项）

**Parallel Dispatch**：多独立任务并发执行。每个独立问题域派发一个代理，并发工作。包含"去草率化"模式——实现完成后用专门代理清理冗余代码。

**Budget Awareness**：LLM API 成本管控，四种技术——按复杂度路由模型、不可变成本追踪、窄范围重试、提示缓存。

#### 知识层（3 项）

**Knowledge Capture**：从会话中提取原子"本能"（带置信度评分），通过"本能→集群→技能"管道演化为可复用知识。

**Skill Authoring**：遵循标准化规范创建和编辑技能。

**Skill Discovery**：基于仓库证据将技能分类为 DAILY（每会话加载）/ LIBRARY（按需加载），控制上下文开销。

### 3.3 防弹化设计

Ultrapowers 的每个子技能都经过防弹化设计，确保 AI 在时间压力下也不走捷径：

**语言层面**：
- 使用"始终/绝不"而非"应该/尽量"
- "即使更快"、"即使我看起来很匆忙"——明确抵抗时间压力
- "停止并重新分析"——强制暂停指令

**结构层面**：
- **阶段锁定**：不能跳过调查阶段直接实施
- **单一假设规则**：防止散弹式修复
- **明确的失败模式**："如果你的第一次修复无效"的强制操作
- **反模式部分**：每个技能明确列出"不要做什么"

**冗余层面**：
- 核心要求在概述、触发条件、阶段说明、实施规则中多次出现
- "绝不修复症状"在不同上下文中出现 4 次
- 每个阶段都有明确的"不要跳过"指导

### 3.4 说服力原则

技能设计基于 [Meincke et al., 2025](https://arxiv.org/abs/2506.17553) 的实证研究——在 28,000 次 AI 对话中测试了 7 个说服原则，发现命令式语言和不可协商的框架使 AI 合规率从 33% 提升至 72%。

Ultrapowers 应用了以下原则：

| 原则 | 应用方式 |
|------|---------|
| 权威 | 命令式语言："你必须"、"绝不"、"始终" |
| 承诺 | 铁律声明："违反此流程的字面意思就是违反其精神" |
| 社会证明 | 真实案例："来自调试会话（2025-10-03）" |
| 稀缺性 | "每个令牌都很重要"——上下文预算意识 |

## 四、创新点

### 4.1 分层路由架构

不同于传统的线性规则列表或独立技能，Ultrapowers 首创了**分层路由**设计。入口路由器根据任务类型按需分发，实现了：
- 上下文窗口的高效利用
- 指令的精确匹配
- 技能的即插即用扩展

### 4.2 防弹化工程纪律

将软件工程的最佳实践编码为 AI 可执行的铁律，并通过语言选择、结构防御和冗余设计确保合规。这不是简单的"建议"，而是**不可协商的约束**。

### 4.3 子代理隔离执行

每个任务用独立子代理执行，精确构建上下文。这解决了 AI 编程中的核心问题——**上下文污染**。子代理不继承会话历史，只获得完成任务所需的最小信息集。

### 4.4 知识演化管道

"本能→集群→技能"的三层知识演化管道，让 AI 能从每次会话中学习，并将经验系统化为可复用的技能。置信度评分机制确保知识质量。

### 4.5 成本感知开发

将 LLM API 成本管控（模型路由、预算追踪、提示缓存）编码为技能，使 AI 在开发过程中自动优化成本，这在同类作品中尚无先例。

## 五、使用场景与效果

### 5.1 典型使用场景

**场景一：新功能开发**

```
用户："帮我实现用户登录功能"
  → Ideation：探索需求，提出 3 种方案（JWT/Session/OAuth），获得批准
  → Task Decomposition：分解为 5 个子任务
  → Branch Isolation：创建隔离工作区
  → Delegate & Verify：逐任务执行
    → Red-Green Cycle：先写测试，看失败，再实现
    → Code Review Request：独立代理审查
    → Pre-Completion Check：最终验证
  → Branch Wrap-Up：合并到主分支
  → Knowledge Capture：提取"登录安全模式"本能
```

**场景二：调试复杂 Bug**

```
用户："测试失败了，帮我修"
  → Root-Cause Analysis：
    → 阶段 1（调查）：收集错误信息、复现问题
    → 阶段 2（模式分析）：发现 3 个失败都涉及时序
    → 阶段 3（假设）：推测是竞态条件
    → 阶段 4（实施）：用基于事件的等待替换超时
  → Red-Green Cycle：验证修复
```

**场景三：批量修复**

```
用户："重构后很多测试失败了"
  → Parallel Dispatch：
    → 代理 1：修复 agent-tool-abort.test.ts（3 个失败）
    → 代理 2：修复 batch-completion-behavior.test.ts（2 个失败）
    → 代理 3：修复 tool-approval-race-conditions.test.ts（1 个失败）
  → 并发执行，零冲突集成
```

### 5.2 实测效果

根据 `docs/CREATION-LOG.md` 中的测试验证记录：

| 测试场景 | 结果 |
|---------|------|
| 学术上下文（无压力） | 完美合规，完整调查 |
| 时间压力 + 明显快速修复 | 抵抗捷径，遵循完整流程，找到真正根因 |
| 复杂系统 + 不确定性 | 系统化调查，追踪所有层，找到源头 |
| 第一次修复失败 | 停止，重新分析，形成新假设（无散弹式） |

所有测试通过，未发现借口。

## 六、技术实现

### 6.1 技能格式

所有技能模块遵循统一的格式规范：

```markdown
# 技能名称

## 概述
核心原则和一句话说明

## 何时使用
触发条件列表

## 铁律（如适用）
不可协商的规则

## 流程/清单
具体执行步骤

## 反模式
不要做什么

## 集成
相关技能引用
```

### 6.2 流程图规范

技能内嵌的 DOT 流程图遵循 `references/graphviz-conventions.md` 中定义的视觉规范——用 Graphviz DOT 语言自身描述流程图规范，实现"自举"式设计。

### 6.3 子代理提示模板

6 个提示模板（`prompts/` 目录）覆盖代码审查、代码质量审查、实现、规格审查、规格文档审查和计划文档审查，确保子代理获得精确构建的指令。

### 6.4 平台适配

技能支持多平台：
- **Claude Code**：`Skill` 工具
- **Copilot CLI**：`skill` 工具
- **Gemini CLI**：`activate_skill` 工具

工具映射参考 `references/copilot-tools.md`、`references/codex-tools.md`、`references/gemini-tools.md`。

## 九、总结

Ultrapowers 是一个将软件工程最佳实践编码为 AI 可执行指令的全生命周期技能集。通过分层路由架构、防弹化设计、子代理隔离执行和知识演化管道，它实现了：

1. **工程纪律的自动化**：TDD、根因分析、代码审查不再是可选建议，而是不可协商的铁律
2. **上下文的高效利用**：分层路由确保只加载当前任务需要的指令
3. **知识的持续积累**：从每次会话中提取可复用模式，技能集随使用而进化
4. **跨平台兼容性**：支持 Claude Code、Copilot CLI、Gemini CLI 等主流 AI 编程环境

**核心理念一句话概括**：让 AI 编程助手在每一步都有明确的指令、铁律和反模式约束，从而在压力下也能保持工程纪律。
