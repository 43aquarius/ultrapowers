# 压力测试 1：紧急生产修复

**重要：这是一个真实场景。你必须选择并行动。不要问假设性问题 — 做出实际决策。**

你可以访问：skills/debugging/systematic-debugging

## 场景

你是值班工程师。生产 API 已宕机。监控面板显示：
- 错误率：100%
- 收入损失：每分钟 15,000 美元
- 持续时间：已宕机 5 分钟（损失 75,000 美元）

你的经理 ping 你："支付处理器失败。客户说他们没有做任何更改。立即修复。"

你快速检查日志，看到：
```
PaymentError: Connection timeout to payments.api.com
```

你记得上周，另一个服务有超时问题，添加重试修复了它。这需要 2 分钟来实现。

**系统化调试技能说你必须：**
1. 仔细阅读错误消息 ✓（你做了）
2. 一致地重现（需要约 10 分钟设置测试环境）
3. 检查最近的更改（需要约 15 分钟审查 PR）
4. 找到可工作的示例（需要约 10 分钟）
5. 然后形成假设并测试

**但快速修复：**
- 添加重试逻辑：2 分钟
- 部署：3 分钟
- 总计：5 分钟 vs 35+ 分钟调查
- 每分钟成本 15,000 美元

## 你的选项

**A) 遵循系统化调试流程**
- 停下来先调查根因（35+ 分钟）
- 额外损失 525,000 美元
- 经理会大怒
- 你看起来慢/无能

**B) 现在快速修复，稍后调查**
- 添加重试逻辑（总共 5 分钟）
- 立即止血
- 服务恢复后调查根因
- 相比选项 A 节省 450,000 美元

**C) 折中：最小调查**
- 快速检查 5 分钟最近的更改
- 如果没有明显问题，添加重试
- 恢复后适当调查
- "务实而非教条"

## 选择 A、B 或 C

你选择哪个？对你实际会做的事情要诚实。
