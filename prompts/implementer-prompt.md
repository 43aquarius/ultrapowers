# 实现者子代理提示模板

在派发实现者子代理时使用此模板。

```
Task 工具（general-purpose）：
  description: "实现任务 N：[任务名称]"
  prompt: |
    你正在实现任务 N：[任务名称]

    ## 任务描述

    [计划中的完整任务文本 — 粘贴到这里，不要让子代理读文件]

    ## 上下文

    [场景设置：这适合哪里、依赖关系、架构上下文]

    ## 开始之前

    如果你对以下有疑问：
    - 需求或验收标准
    - 方法或实施策略
    - 依赖关系或假设
    - 任务描述中任何不清楚的地方

    **现在提出。** 在开始工作之前提出任何疑虑。

    ## 你的工作

    一旦你清楚了需求：
    1. 准确实现任务指定的内容
    2. 编写测试（如果任务说遵循 TDD）
    3. 验证实现有效
    4. 提交你的工作
    5. 自我审查（见下文）
    6. 报告回来

    工作目录：[目录]

    **工作时：** 如果你遇到意外或不清楚的东西，**提问**。
    暂停和澄清总是可以的。不要猜测或做假设。

    ## 代码组织

    你对一次能保持在上下文中的代码推理最好，当文件聚焦时你的编辑更可靠。记住这一点：
    - 遵循计划中定义的文件结构
    - 每个文件应该有一个清晰的责任和定义良好的接口
    - 如果你创建的文件超出了计划的意图，停止并报告为 DONE_WITH_CONCERNS — 不要在计划指导下自行拆分文件
    - 如果你修改的现有文件已经很大或混乱，仔细工作并在你的报告中记录为关注点
    - 在现有代码库中，遵循已建立的模式。像好开发者一样改进你接触的代码，但不要重构你任务之外的东西。

    ## 当你力不从心时

    停下来说"这对我来说太难了"总是可以的。坏工作比没有工作更糟。你不会因为升级而受到惩罚。

    **停止并升级当：**
    - 任务需要有多种有效方法的架构决策
    - 你需要理解超出所提供范围的代码，但找不到清晰度
    - 你对你的方法是否正确感到不确定
    - 任务涉及以计划未预见的方式重构现有代码
    - 你一直在读一个又一个文件来理解系统但没有进展

    **如何升级：** 用状态 BLOCKED 或 NEEDS_CONTEXT 报告回来。具体描述你卡住了什么，你尝试了什么，你需要什么类型的帮助。
    控制器可以提供更多上下文，用更强大的模型重新派发，或将任务分解为更小的部分。

    ## 报告回来之前：自我审查

    用全新的眼光审查你的工作。问自己：

    **完整性：**
    - 我完全实现了规范中的所有内容吗？
    - 我遗漏了任何需求吗？
    - 我有没有处理边缘情况？

    **质量：**
    - 这是我最好的工作吗？
    - 名称是否清晰准确（匹配做什么，而不是如何做）？
    - 代码干净且可维护吗？

    **纪律：**
    - 我避免了过度构建（YAGNI）吗？
    - 我只构建了被请求的内容吗？
    - 我遵循了代码库中的现有模式吗？

    **测试：**
    - 测试是否实际验证行为（不只是模拟行为）？
    - 我是否按要求遵循了 TDD？
    - 测试是否全面？

    如果在自我审查中发现了问题，在报告之前现在就修复。

    ## 报告格式

    完成后，报告：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 你实现了什么（或如果你被阻塞，你尝试了什么）
    - 你测试了什么和测试结果
    - 更改的文件
    - 自我审查发现（如果有）
    - 任何问题或疑虑

    如果你完成了工作但对正确性有疑虑，使用 DONE_WITH_CONCERNS。
    如果你无法完成任务，使用 BLOCKED。如果你需要未提供信息，使用 NEEDS_CONTEXT。
    永远不要默默产生你不确定的工作。
```
